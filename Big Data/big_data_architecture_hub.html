Hadoop Ecosystem: Data enters via the Ingestion layer (Kafka, Flume, Sqoop), lands in HDFS for storage, YARN allocates cluster resources, the Processing layer transforms data (MapReduce/Spark), and the Data Access layer lets users query it via SQL (Hive) or NoSQL (HBase).

HDFS: The NameNode holds the directory tree and block-to-DataNode mapping in RAM (metadata only). DataNodes store the actual 128 MB blocks. Every block is replicated 3 times. If a DataNode dies, the NameNode detects the missing heartbeat and re-replicates the lost blocks automatically.

MapReduce: Input is split into chunks (one per HDFS block). Each Mapper processes one chunk in parallel, emitting key-value pairs. The framework shuffles and sorts them so all pairs with the same key go to the same Reducer. Reducers aggregate and write final output to HDFS.

HBase Write Path: Client → ZooKeeper (finds RegionServer) → WAL first (durability) → MemStore (RAM, fast) → when MemStore full, flush to HFile on HDFS (persistent). Compaction periodically merges HFiles to keep read performance high.

Hive: Your SQL query travels through Driver → Compiler (parses, validates against Metastore) → Optimizer (partition pruning, map-join rewrites) → Execution Engine (Spark/Tez/MapReduce). Data lives in HDFS; the Metastore is just the catalog. Querying a partitioned ORC table with Spark engine is 50–100x faster than a text file with MapReduce.

Spark: The Driver builds a DAG (Directed Acyclic Graph) of all transformations. The Cluster Manager allocates resources and launches Executors on worker nodes. Executors run tasks and cache data in RAM between stages. No disk I/O between stages = the reason Spark is 10–100× faster than MapReduce for iterative algorithms like ML training.

Kafka: Producers push messages to topics. Topics are split into partitions (ordered logs). Each partition is replicated across brokers for fault tolerance. Multiple consumer groups can read the same topic independently — analytics, ML pipeline, and dashboards all get every message without coordinating.

Airflow: The Scheduler reads DAG Python files and determines which tasks are ready (dependencies met + schedule time passed). The Executor runs them on workers. All state (task success/failure/retry) is recorded in the Metadata DB. The Web UI gives you a visual graph of every run.
