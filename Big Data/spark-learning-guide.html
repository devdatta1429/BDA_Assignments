Chapter 01

Spark Architecture

Think of Spark like a restaurant. One Head Chef gives orders. Multiple Cooks actually cook the food. That's exactly how Spark works.

Spark uses a master/slave (driver/worker) architecture. There is one central coordinator called the Driver, and many distributed workers called Executors. They all run in their own Java processes.

Driver Program
The main program. It converts your code into tasks and schedules them. Like the Head Chef reading the menu.
SparkContext
The entry point of any Spark app. Created inside the Driver. Without it, nothing works. It's how your code connects to the cluster.
Cluster Manager
Manages cluster resources. It decides which worker gets which task. YARN is the cluster manager used in today's class.
Executor
A process on a Worker Node that actually runs your tasks. Started at the beginning of a Spark application and lives for its entire lifetime.
Worker Node
The physical/virtual machine where Executors run. Each worker reports its available resources to the Cluster Manager.
Block Manager
Part of each Executor that provides in-memory storage for cached RDDs.

Application Execution Flow — Step by Step

1

SparkContext is Created

Your program starts and creates a SparkContext instance. Only now can your program be called a "Driver".

2

Driver Asks for Resources

The Driver asks the Cluster Manager to launch Executors on Worker Nodes.

3

Cluster Manager Launches Executors

Executors are started on Worker Nodes. They register themselves with the Driver directly.

4

Driver Sends Tasks

The Driver runs through your code. When it hits transformations and actions on RDDs, it sends tasks to Executors.

5

Executors Run & Return Results

Executors run the tasks on their partitions and send results back to the Driver.

6

Fault Tolerance in Action

If a Worker crashes, its tasks are sent to different Executors. Spark even uses "speculative execution" — if a node is just slow, it launches a copy of the task on another node and takes whichever finishes first.

💡 Remember

When you type pyspark in the terminal, a SparkContext is automatically created for you as sc. That's why in today's lab we could directly write rdd1 = sc.textFile(...)

Chapter 02

RDDs — Resilient Distributed Datasets

An RDD is the fundamental data structure in Spark. Think of it as a giant list that's been split across many computers.

RDD stands for Resilient Distributed Dataset. Let's break that down:

Resilient
Fault-tolerant. If a partition of your data is lost, Spark can re-compute it from scratch using the lineage (history of transformations).
Distributed
The data is split across multiple worker nodes. Each piece (partition) lives on a different machine and is processed in parallel.
Dataset
Simply a collection of data — could be records, words, numbers, rows — anything.
Immutable
You cannot change an RDD. Every transformation creates a NEW RDD. This is why Spark cannot return incomplete output.
🔑 Key Concept: Lazy Evaluation

Spark is lazy. When you write a transformation (like map or filter), Spark doesn't actually run anything. It just takes notes. Only when you call an Action (like collect() or saveAsTextFile()) does Spark actually execute everything.

Two Types of RDD Operations

Every operation you do on an RDD is one of two types:

Transformation

Creates a NEW RDD from an existing one. Does NOT execute immediately.

Examples: map(), filter(), flatMap(), reduceByKey(), repartition()

Action

Triggers actual execution and returns a result. This is when Spark actually does the work.

Examples: collect(), count(), take(n), saveAsTextFile()

📌 Today's trainer said: "Till Action is not part of the code, none of the coding will get executed." This is Lazy Evaluation in action.

DAG — Directed Acyclic Graph

As you write transformations, Spark builds a DAG — a roadmap of all the steps. Think of it like Google Maps planning a route before you start driving. The DAG Scheduler reads this map and divides work into stages and tasks when an action is finally triggered.

Chapter 03

Key Transformations

These are the operations you used today in the Word Count and Top Selling Products programs.

flatMap — Split & Flatten

Used to split each line into individual words. Unlike map() which gives one output per input, flatMap() can give multiple outputs per input and flattens them all into one list.

flatMap Example
# Input: "river car river" is ONE element
rdd3 = rdd2.flatMap(lambda a : a.split(" "))

# Output: "river", "car", "river" are THREE separate elements
# The list is FLATTENED — no nested arrays
FLATMAP VISUAL
Input
→
"river car river"
"car car bear"
flatMap
→
split(" ") each line → flatten all results
Output
→
"river", "car", "river", "car", "car", "bear"

map — Transform Each Element

Applies a function to every element and returns one output per input. Used to convert each word into a key-value pair like ('car', 1).

map Example — Creating Key-Value Pairs
rdd4 = rdd3.map(lambda a : (a, 1))

# "car"   →  ('car', 1)
# "river" →  ('river', 1)
# "bear"  →  ('bear', 1)

reduceByKey — Group & Aggregate by Key

Groups all values with the same key and applies a function to combine them. This is what causes a shuffle — data moves across the network so that all records with the same key land on the same partition.

reduceByKey Example — Word Count
rdd5 = rdd4.reduceByKey(lambda a, b : (a + b))

# ('bear', 1) + ('bear', 1) + ('bear', 1) ... → ('bear', 6)
# ('car', 1) + ('car', 1) + ... → ('car', 7)
# ('river', 1) + ... → ('river', 9)

# Result:
# ('bear', 6)  ('car', 7)  ('river', 9)  ('dear', 1)
💡 reduceByKey vs SQL

reduceByKey is like SQL's GROUP BY with SUM(). In today's class: select prod, sum(amt) from orders group by prod was equivalent to creating key-value pairs of (product, amount) then doing reduceByKey.

repartition — Change Number of Partitions

You can manually set how many partitions an RDD should have. More partitions = more parallelism. Less partitions = less overhead.

repartition Example — From Today's Lab
rdd1 = sc.textFile("/user/cddacmaylab013/data")
rdd1.getNumPartitions()   # Returns: 3 (3 files = 3 blocks)
sc.defaultParallelism     # Returns: 2 (system has 2 cores)

rdd2 = rdd1.repartition(2)  # Now 2 partitions to match 2 cores
rdd2.getNumPartitions()   # Returns: 2 ✓

sortBy — Sort the Data

Used in the Top Selling Products program to sort products by total sales in descending order.

sortBy Example — Top Selling Products
rdd4 = rdd3.sortBy(lambda a : -a[1])  # negative = descending order

# Result (top items):
# ('Yoga & Pilates', 47804.93)
# ('Swing Sets', 47204.14)
# ('Lawn Games', 46828.43)
Chapter 04

Actions — Trigger Execution

Actions are what actually make Spark DO something. Without an action, your transformations are just a plan sitting on paper.

collect()
Brings ALL data from all partitions back to the Driver as a Python list. Use carefully on large datasets — can crash the Driver if data is huge!
count()
Returns the total number of elements in the RDD. Simple but triggers full computation.
take(n)
Returns only the first n elements. Safer than collect() for previewing large datasets.
saveAsTextFile()
Saves the RDD to HDFS (or disk) as text files. One file per partition. Used at the end of both programs today.
coalesce(n)
Reduces the number of partitions without a full shuffle. Used to merge output files before saving. Unlike repartition, it only reduces — never increases.
⚠️ Important Distinction

Transformations return a new RDD (lazy — no execution).
Actions return a value or write to storage (eager — triggers execution).

This is why in the Word Count code, typing rdd3 = rdd2.flatMap(...) was instant — but rdd3.collect() took time to run.

Chapter 05

Partitions — How Data is Split

A partition is one chunk of your dataset. More partitions = more parallel processing. Understanding partitions is key to understanding Spark's speed.

When you load a file into Spark, it gets divided into partitions. In HDFS, data is stored in blocks (default 128MB each). Each block becomes one partition in Spark.

📂 From Today's Lab

The data folder had 3 files (3 blocks in HDFS), so rdd1 had 3 partitions. The system had 2 CPU cores, so we repartitioned to 2 to match the cores for better efficiency.

Hash Partitioning

When you use reduceByKey(), Spark decides which partition a key goes to using hash partitioning:

Hash Partitioning Formula
# From today's class:
hash_value('bear') = 10
num_partitions = 2

partition_number = hash_value % num_partitions
                 = 10 % 2
                 = 0   # 'bear' goes to Partition 0

# This ensures ALL 'bear' records land in the same partition
# so they can be counted together!
Range vs Hash Partitioning

Hash Partitioning: Uses hash(key) % num_partitions. Good for grouping by key (like reduceByKey).
Range Partitioning: Divides keys into sorted ranges. Good for sorted output (like sortBy).

Jobs, Stages & Tasks

When Spark runs your code, it breaks everything into a hierarchy: one Job → multiple Stages → many Tasks.

The Hierarchy

JOB — Triggered by one Action (e.g. collect() or saveAsTextFile())
STAGE 0 — ShuffleMapStage (before data shuffling: flatMap → map)
Task 1
Partition 0
Task 2
Partition 1
STAGE 1 — ResultStage (after shuffle: reduceByKey → Action)
Task 1
Partition 0
Task 2
Partition 1
Job
One job = one action call. Every time you call collect(), count(), saveAsTextFile(), etc., Spark creates one job. A job is a series of stages.
Stage
A stage is created whenever data needs to be shuffled across the network. reduceByKey() causes a shuffle → creates a stage boundary.
Task
The smallest unit of work. One task runs on ONE partition of data on ONE core of a worker node. 2 partitions = 2 tasks per stage.
ShuffleMapStage
A stage that produces data FOR a shuffle operation. Its output feeds into the next stage.
ResultStage
The FINAL stage in a job. It runs the action and returns the final result to the Driver.
Pipelining
Within a single stage, transformations are chained together and processed sequentially on each partition without extra shuffling.
Rule: When does a new Stage start?

A new stage begins whenever there is a data shuffle — when records must move from one partition to another across the network. This happens with operations like reduceByKey(), groupByKey(), join(), sortBy().

No shuffle = same stage. flatMap → map → collect (no shuffle) = ONE stage.
