Part 1

HDFS — Hadoop Distributed File System

How Hadoop stores terabytes of data across hundreds of cheap machines
Section 1.1

What is HDFS?

Imagine you have a 1 TB movie file. No single laptop can store it. HDFS solves this by cutting that file into 128 MB chunks and spreading them across hundreds of machines — but to you it still looks like one normal file.

🏛️ Library Analogy

Think of HDFS like a huge city library. There is one librarian (NameNode) who knows which shelf every book is on. There are many shelves (DataNodes) that actually store the books. When you (the Client) want a book, you ask the librarian first, then go directly to the shelf.

Block
A fixed 128 MB chunk of a file. A 400 MB file becomes 3 full blocks + 1 small block. Each block lives on a DataNode's local disk.
Replication
Every block is copied 3 times by default. If one machine dies, 2 other copies still exist. You never lose data.
Namespace
The directory tree of all files and folders — like your computer's folder structure. Stored entirely in the NameNode's RAM.
Metadata
Information ABOUT your data — file names, sizes, permissions, which blocks belong to which file. NOT the actual data itself.

Section 1.2

The NameNode — Brain of HDFS

The NameNode is the single master server. It lives entirely in RAM for speed and handles ALL metadata operations — but never touches actual data bytes.

🔑 Critical Fact

The NameNode stores ONLY metadata — file names, permissions, block locations. It does NOT store your actual data. Not a single byte of your file content lives on the NameNode. A large cluster NameNode might hold 200 GB of metadata in RAM.

Sub-component: FSImage — The Photograph

FSImage is a frozen snapshot of the entire file system at a point in time — like a photograph taken yesterday. When the NameNode restarts, it loads this photo first to know what files exist. Without it, restarting = losing everything.

Sub-component: EditLog — The Diary

EditLog is a transaction journal — like a diary that records every single change. Every time you create a file, delete a folder, or rename something, it gets written to the EditLog on disk.

When the NameNode crashes and restarts: it loads the FSImage (old photo) + replays the EditLog (diary entries since the photo) = reconstructs the exact current state. Clever!

📷 Analogy

FSImage = a photograph of your house. Shows everything as it was at that moment.
EditLog = a diary of changes since the photo. "Added a chair, painted the wall, removed a table."
Restart = load the photo + replay the diary → exact current state.

⚠️ The EditLog Problem

Over time the EditLog grows huge — millions of entries. On restart, replaying all of them takes forever. This is why the Standby NameNode (or old Secondary NameNode) periodically merges FSImage + EditLog into a fresh, compact FSImage. This is called checkpointing.

Section 1.3

Standby NameNode — True Hot Backup

Introduced in Hadoop 2.x. Unlike the old Secondary NameNode, the Standby is a real backup — if the Active NameNode dies, it takes over within seconds with ZERO data loss.

The Standby NameNode is like a co-pilot on an airplane. The pilot (Active NameNode) is flying the plane. The co-pilot is watching every move and mirroring every control input in real time. If the pilot suddenly collapses, the co-pilot can immediately take over — because they already know exactly what was happening.

In-Memory Mirror
The Standby keeps a complete copy of the full file system namespace in its own RAM. It applies every edit from JournalNodes in real time — always only milliseconds behind Active.
Checkpoint Service
In HA mode, the Standby takes over checkpointing duty. It periodically merges FSImage + EditLog into a new compact FSImage and sends it back to the Active. Keeps startups fast.
⚠️ Secondary NameNode ≠ Backup!

The old Secondary NameNode (Hadoop 1.x) was NOT a hot backup. It only did checkpointing. If the Active NameNode crashed, the Secondary could NOT take over. It's legacy — replaced by the true Standby NameNode in HA setups.

Section 1.4

JournalNode Quorum — Shared EditLog

In HA mode, both Active and Standby NameNodes need to share the same EditLog. JournalNodes are the infrastructure that makes this shared EditLog possible — and fault-tolerant.

Think of JournalNodes as a group of notaries. When the Active NameNode makes a change (creates a file, deletes a folder), it must get that change confirmed by a majority of notaries before proceeding. The Standby NameNode continuously reads these confirmed records and updates its own copy.

💡 Why Always Odd Numbers?

JournalNodes (and ZooKeeper) are always deployed in odd numbers (3, 5, 7...) because of majority voting. With 3 nodes, losing 1 still leaves a 2-node majority — cluster keeps running. With 4 nodes, losing 2 means you're stuck at exactly 50-50 — no majority possible. Odd numbers guarantee a majority can always be formed.

1

Active NameNode writes an edit

Every change (create file, delete folder) is written to ALL JournalNodes simultaneously. The Active waits for at least 2 out of 3 to confirm.

2

Standby reads continuously

The Standby NameNode continuously reads (tails) these edit segments from the JournalNodes, replaying each one to stay in sync with Active.

3

Fault tolerance

If one JournalNode dies, the other two still form a majority. The system keeps running. Only if 2+ JournalNodes die does the cluster halt writes.

Section 1.5

ZooKeeper & ZKFC — Automatic Failover

ZooKeeper ensures only ONE NameNode is Active at any time. ZKFC is the agent that monitors the NameNode and uses ZooKeeper to trigger automatic failover.

🔑 Analogy — The Hotel Key Card

ZooKeeper is like a hotel that gives out only one master key card. The Active NameNode holds this card. As long as it's alive, it keeps using the card. If it dies, the card is automatically deactivated and a new one is issued to the Standby NameNode — which can now open all the doors.

The Split-Brain Problem

Without ZooKeeper, imagine the Active NameNode becomes slow due to network issues. The Standby thinks it's dead and promotes itself to Active. Now both NameNodes think they're Active and both start accepting writes. This is called Split-Brain — and it leads to namespace corruption, two conflicting versions of truth.

ZooKeeper prevents this with a simple rule: only the NameNode holding the ZooKeeper lock can be Active. Only one can hold the lock at a time.

ZKFC — The Watchdog

ZKFC (ZooKeeper Failover Controller) is a small daemon that runs alongside each NameNode. It has three jobs:

Health Monitor
Continuously sends health checks to the local NameNode. If NameNode stops responding, ZKFC acts.
ZK Session Manager
The Active NameNode's ZKFC holds the ZooKeeper lock. If the NameNode dies, the session expires and the lock is released.
Fencing
Before promoting Standby to Active, ZKFC forcibly kills or disconnects the old Active. This prevents split-brain. Only then does Standby take over.
Section 1.6

DataNodes — The Storage Workers

DataNodes are the actual storage machines. A production cluster can have 50 to 5,000 DataNodes, each a cheap commodity server with lots of disk. They store blocks and serve them on demand.

Block Storage
Files are split into 128 MB blocks. A 400 MB file = 3 blocks of 128 MB + 1 block of 16 MB. Each block stored as a plain file on local disk. DataNode doesn't know about the whole file — just its own blocks.
Block Checksum
When writing a block, a CRC-32C checksum is computed and stored. When reading, the checksum is recomputed and compared. If they don't match = corrupt block → reported to NameNode → another replica is fetched.
Heartbeat
Every 3 seconds, each DataNode tells the NameNode "I'm alive" + current disk space, CPU load. No heartbeat for 10 minutes = NameNode marks it dead and starts re-replicating its blocks.
Block Report
Every hour, each DataNode sends a complete list of ALL its blocks to the NameNode. Lets NameNode verify the block map is accurate and detect anything missing.
Block Scanner
Background thread on every DataNode. Continuously re-reads and rechecks blocks on disk. Finds silent corruption (disk bit-flips) before a client reads them.
Replication Pipeline
When writing, the client sends data to DataNode 1 → DN1 forwards to DN2 → DN2 forwards to DN3. All 3 copies written simultaneously. ACKs travel back: DN3→DN2→DN1→Client.

Rack Awareness — Smart Replica Placement

Servers in a data centre are grouped in racks (20–40 servers per rack share one network switch). HDFS places 3 replicas like this:

🗄️ Rack Awareness Rule

2 replicas → same rack (fast intra-rack network speed)
1 replica → different rack (protection if the whole rack loses power or its switch dies)

Example: Block A → Copy 1 on Rack 1 Server 1, Copy 2 on Rack 1 Server 3, Copy 3 on Rack 2 Server 7.

Section 1.7

HDFS Read & Write — Step by Step

Understanding exactly what happens when you save or read a file from HDFS.

📝 Writing a File to HDFS

1

Client contacts NameNode

Client says "I want to create /data/sales.csv". NameNode checks permissions, creates metadata entry, tells client which DataNodes to write each block to.

2

Replication Pipeline is set up

Client streams data to DataNode 1. DN1 forwards to DN2. DN2 forwards to DN3. All 3 copies written simultaneously as data flows through the chain.

3

ACKs travel back

DN3 confirms → DN2 confirms → DN1 confirms → Client. Only when the client receives all ACKs is the block considered safely written.

4

NameNode is updated

After all blocks written, client tells NameNode to close the file. NameNode records the final block-to-DataNode mapping and writes to EditLog.

📖 Reading a File from HDFS

1

Client requests file metadata

Client says "I want /data/sales.csv". NameNode returns list of block locations — for each block, which DataNodes hold copies, sorted by network distance (closest first).

2

Client reads directly from DataNodes

No NameNode involvement! Client connects directly to the nearest DataNode per block and streams data. NameNode is NOT a bottleneck for data transfer.

3

File is reassembled

As blocks arrive in order, the client stitches them together to reconstruct the complete original file.

YARN — Yet Another Resource Negotiator

The operating system of the Hadoop cluster — manages CPU and RAM for all applications
Section 2.1

What is YARN?

In Hadoop 1.x, MapReduce did two very different things: managed cluster resources AND ran data processing. YARN separated these into two clean layers — making Hadoop a platform for ANY framework, not just MapReduce.

🏭 Factory Floor Analogy

HDFS = the warehouse (stores raw materials)
YARN = the factory floor manager (decides which machines do which work, allocates equipment)
MapReduce / Spark / Flink = the assembly line instructions (the actual work being done)

YARN doesn't care what kind of product you're making — it just provides machines and equipment. That's why you can run Spark, Flink, and MapReduce on the same cluster simultaneously.

Section 2.2

ResourceManager — The Cluster's Boss

The single master of YARN. Has complete authority over every CPU core and every GB of RAM across all worker machines. It does NOT process data — only manages who gets resources.

Sub-component: Scheduler — Who Gets What

The Scheduler looks at all pending resource requests and decides which application gets which resources on which machines. It is completely pluggable — 3 policies available:

FIFO Scheduler
First come, first served. Job 1 gets everything it needs. Job 2 waits until Job 1 finishes. Simple but terrible for shared clusters — one huge job can block everything for hours.
Capacity Scheduler
Divide cluster into queues — "Production gets 60%, Dev gets 40%". Each team submits to their queue. Guaranteed share. Unused capacity can be borrowed. Default in most production clusters.
Fair Scheduler
All apps get equal share over time. 1 job = 100%. 2 jobs = 50% each. One job finishes = other reclaims 100%. Best for clusters with many short interactive queries.
🪑 Capacity Scheduler Example

Imagine an office building with 100 desks. The Data Science team is guaranteed 60 desks and the Engineering team is guaranteed 40 desks. If Data Science only uses 40 desks today, Engineering can temporarily borrow 20 extra desks. When Data Science needs those 20 back, Engineering gradually returns them.

Sub-component: ApplicationsManager — Job Lifecycle

Accepts job submissions, validates them, and asks the Scheduler for one small container to start the ApplicationMaster. Once AM is running, monitors its health. If the AM crashes, re-launches it. When the job finishes, cleans up.

Section 2.3

ApplicationMaster — One Per Job

The most elegant design decision in YARN. Every single application (MapReduce job, Spark job, etc.) gets its own dedicated ApplicationMaster process — a mini-manager just for that one job.

🏗️ Project Manager Analogy

The ResourceManager is like a company's CEO — manages all resources across the whole company. The ApplicationMaster is like a Project Manager hired for one specific project. The PM negotiates resources from the CEO, manages the workers on that project, handles problems, and reports progress. When the project ends, the PM goes home.

Resource Negotiator
AM asks the ResourceManager: "I need 50 containers, each with 4 GB RAM and 2 CPU cores, preferably on machines that have the data I need." Gets back a list of available containers.
Task Monitor
AM monitors every task running in its containers. If a task fails (machine crashes, memory overflow), AM requests a replacement container and re-runs the task. RM doesn't handle individual task failures — only the AM does.
Progress Reporter
AM continuously reports job progress (% complete, tasks running/failed/succeeded) back to the RM. When the job finishes, AM releases all containers and shuts itself down.
💡 Why this design is genius

The ResourceManager never needs to know how MapReduce or Spark work internally. Each framework brings its own ApplicationMaster and manages its own tasks. YARN just provides raw containers. This is what makes YARN truly generic — you can run ANY distributed computing framework on top of it.

Section 2.4

NodeManager — The Worker Agent

A NodeManager daemon runs on EVERY worker machine. It manages that machine's resources and launches containers when instructed by the ResourceManager and ApplicationMaster.

Container
A guaranteed slice of one machine's CPU + RAM. Like a private booth in an office — reserved just for you, nobody else can use those resources while your task is running. One JVM task runs inside each container.
Container Executor
Actually launches the container JVM process on the OS. In secure clusters, uses LinuxContainerExecutor to run the container under the submitting user's Unix account — your job runs as you, not as "hadoop" system user.
Node Health Checker
Monitors the whole machine's health — is disk full? Is YARN's local scratch space available? If node is unhealthy, reports to RM which stops sending new containers to that machine.
Log Aggregator
After a container finishes, uploads all its logs (stdout, stderr, app logs) from local disk to HDFS. Without this, logs vanish when the container is cleaned up. Run "yarn logs -applicationId <id>" to view them later.
Localisation Service
Before a container runs, downloads the JAR files and config files it needs from HDFS to the local disk. Caches them so the same JAR isn't re-downloaded for every container.

Complete YARN Job Lifecycle

1

Job Submission

You run hadoop jar myjob.jar. Client sends JAR + config to ResourceManager's ApplicationsManager.

2

AM Container Allocated

ApplicationsManager asks Scheduler for one small container to run the ApplicationMaster. Scheduler picks a NodeManager with free resources.

3

AM Launched

NodeManager launches AM inside the container. Localisation Service downloads the job JAR first. AM starts up and registers with RM.

4

AM Calculates Resource Needs

AM contacts HDFS to find how many input blocks exist. Sends request to Scheduler: "I need N containers for Map tasks, preferably near the data."

5

Scheduler Grants Containers

Scheduler grants containers across NodeManagers, trying to put Map containers on the same machine as the HDFS data (data locality).

6

Tasks Execute

Map tasks run in parallel. AM monitors all of them. If any fail, AM requests replacement containers. Reduce containers allocated after enough Maps finish.

7

Job Completes

All Reducers finish. AM reports success to RM. AM releases all containers and shuts down. Log Aggregators upload logs to HDFS.

Part 3

MapReduce — Distributed Processing

The programming model that processes terabytes in parallel using just two functions: map() and reduce()
Section 3.1

What is MapReduce?

Imagine counting how many times each word appears in 10 TB of text files. One computer would take days. MapReduce divides the work into thousands of tiny tasks that all run at the same time. You only write two functions.

📚 Library Word Count Analogy

You want to count how many times each word appears in a 1000-page encyclopedia. Instead of reading it yourself:

Map phase: Give one chapter to each of 50 students. Each student writes: "I found 'science' 3 times, 'physics' 2 times..."
Shuffle: Collect all papers, and group them by word — all papers mentioning 'science' go to Student A, all mentioning 'physics' go to Student B.
Reduce phase: Each student adds up the counts for their word. Student A: science = 3+5+2+7 = 17. Done!

You only needed to define what each "student" does — MapReduce handles the coordination.

🎯 Core Philosophy: Move Code to Data

Instead of copying 10 TB of data to one machine to process it, MapReduce sends your tiny map() function to run directly on the machines where the data already lives. This is called data locality — the most important performance principle in Hadoop.

Section 3.2

The Mapper — Processing Each Chunk

One Mapper runs per input split (one HDFS block = 128 MB). All Mappers run in parallel. The Mapper reads raw data, converts it to key-value pairs, and writes sorted intermediate output.

Input Splits — One Mapper Per Split

Before any processing starts, MapReduce divides the input into input splits — logical pointers to chunks of data (one per HDFS block). One Mapper is launched per split. 100 splits = 100 Mappers running simultaneously. You don't set this number — it's automatically determined by the file size.

📌 Split vs Block

An input split is a logical division used by MapReduce (just a pointer: start byte, end byte, location). An HDFS block is physical storage on disk. They usually match in size (128 MB), but they are different concepts.

RecordReader
Reads raw bytes from the HDFS block and converts them into key-value pairs. Default: LineRecordReader reads line by line → (byte_offset, line_text). You can write custom RecordReaders for CSV, JSON, XML, etc.
map() Function
The code YOU write. Receives one key-value pair at a time. Emits zero, one, or many new key-value pairs. Word Count: receives (offset, "hello world") → emits (hello,1) and (world,1). Must be stateless.
Spill Buffer
100 MB circular buffer that collects map() output in memory. When 80% full, a background thread spills the contents to disk as a sorted file. Prevents millions of tiny disk writes. All spill files merged at end.

Combiner — The Mini-Reducer (Optional but Powerful)

The Combiner runs locally on each Mapper machine BEFORE data is sent across the network. It pre-aggregates results to reduce network traffic.

✉️ Combiner Analogy

Imagine you're mailing individual sticky notes to a central office. Without the Combiner: you mail 500 individual notes each saying "hello: 1". With the Combiner: you bundle them locally first → mail one note saying "hello: 500". Same result, 500x less network traffic.

⚠️ When can you use a Combiner?

Only when the operation is commutative and associative — order of combining doesn't change the result.
✅ Safe: sum, count, max, min
❌ NOT safe: average — you cannot average averages and get the correct result.

Partitioner — Who Goes to Which Reducer?

After the Combiner, the Partitioner decides which Reducer will receive each key. Formula: hash(key) % number_of_reducers. This ensures ALL instances of the same key always go to the SAME Reducer — so it can aggregate them all.

Section 3.3

Shuffle & Sort — The Most Expensive Phase

You write zero code for this phase — MapReduce handles it entirely. But it's the most network-intensive operation: all Mapper outputs travel across the cluster network to their assigned Reducers.

📬 Post Office Analogy

After all the students (Mappers) finish writing their word-count notes, imagine a giant postal system that:
1. Collects all the notes from all students across all buildings (Copy phase)
2. Sorts and groups all notes by word — every 'science' note in one pile (Merge-Sort phase)
3. Delivers each pile to the student assigned to count that word (Reducers)

The postal system handles all the logistics — you just define what to do with the sorted pile.

Copy Phase (Shuffle)
Reducers send HTTP requests to Mapper machines and pull their assigned partition. This is the actual "shuffle" — data moving from where it was produced to where it will be consumed. Most network-intensive step.
Merge-Sort Phase
Each Reducer receives many sorted mini-files (one per Mapper). The framework merges all of them into one big sorted stream. After this: (apple,[1,1,1,...]), (banana,[1,1,...]) — all values for same key grouped together.
Spill File Merge
As partitions arrive from Mappers, they're progressively merged on the Reducer's disk. By the time all Mappers finish, the Reducer has one final sorted stream ready for reduce() — without loading everything into RAM at once.
Section 3.4

The Reducer — Final Aggregation

Reducers run one per partition, all simultaneously. Each Reducer receives all the values for its assigned keys (already sorted and grouped by the Shuffle phase) and produces the final output.

reduce() Function
The second function YOU write. Called once per unique key. Receives the key + a lazy Iterator of all values for that key. Word Count: receives (hello, [1,1,1,...]) → emits (hello, 4729).
Iterator (Lazy)
Values arrive one at a time from disk — NOT all loaded into RAM at once. Critical for words appearing billions of times. You can't fit 1 billion values in RAM, but you can process them one by one.
RecordWriter
Writes the final key-value pairs to HDFS as plain text: "key TAB value". Reducer 0 writes part-00000, Reducer 1 writes part-00001, etc. Job complete when all part files are in the output directory.

Data Locality
Running a Mapper on the SAME machine as the HDFS block it reads. Avoids sending 128 MB across the network. YARN's Scheduler tries to honour this by placing Map containers on DataNode machines.
Container (YARN)
A reserved slice of one machine's CPU + RAM. Map tasks, Reduce tasks, and the ApplicationMaster all run inside containers.
Quorum

Fencing
Forcibly disconnecting or killing the old Active NameNode before promoting Standby. Prevents split-brain. Done by ZKFC before any failover.
