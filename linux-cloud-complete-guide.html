Chapter 01 — Linux Fundamentals

What Linux Actually Is

Before you type a single command, you need to understand what you're really talking to — and why Linux became the foundation of almost all modern computing.

📖 THE ORIGIN STORY

In 1991, a Finnish university student named Linus Torvalds was frustrated. He had a PC but couldn't afford a proper Unix operating system, and the free alternatives were either too limited or had restrictive licenses. So he did what programmers do — he built his own.

He posted to a mailing list: "I'm doing a (free) operating system (just a hobby, won't be big and professional like GNU) for 386(486) AT clones." Thirty years later, Linux runs 97% of the world's top supercomputers, all of Android, most of the internet's servers, and every major cloud platform. Not bad for a hobby project.

The Architecture of Linux

Linux is not just a bunch of commands. It has a layered architecture where each layer has a specific job. Understanding this makes everything else click.

Kernel Space vs User Space: This is the most important boundary in Linux. The kernel runs in a privileged mode and can directly access hardware. Your programs run in unprivileged user space — they must ask the kernel for everything via system calls. This is what makes Linux secure and stable: a buggy user program crashes itself, not the whole system.

The GNU Movement — Why Linux is Actually "GNU/Linux"

Linus built the kernel, but a kernel alone is useless — you can't do anything without programs. In the 1980s, Richard Stallman started the GNU project to build a complete free Unix-like system. He created all the tools: gcc (compiler), bash (shell), coreutils (ls, cp, mv, grep…), glibc, and much more.

When Linus built his kernel in 1991, he combined it with GNU tools. The result? A complete, free, usable operating system. The kernel is Linux; the surrounding system is GNU. This is why purists say "GNU/Linux" — it acknowledges both contributions.

The whole thing is distributed under the GPL (GNU General Public License) — a "copyleft" license that says: you can use, modify, and distribute this software freely, but if you distribute a modified version, you must also release your source code. This legal structure is what kept Linux free and ensured the whole community benefits from every improvement.

Linux as a Server — Why It Dominates

You might use Windows or macOS on your laptop. But the server running the website you visit, the database storing your data, the cloud instance running your code — almost certainly runs Linux. Why?

  • ⚡
    Stability: Linux servers routinely run for years without rebooting. Windows servers need regular restarts for updates. A server reboot costs money and availability.
  • 🔒
    Security: Multi-user design baked in from day one. Every file has an owner. Every process runs as a user. The principle of least privilege is the default.
  • 💰
    Cost: Linux is free. Windows Server licenses cost hundreds to thousands of dollars per server. When you have 10,000 servers (like AWS does), this matters enormously.
  • 🛠️
    Automation: Everything is a file or a command. Shell scripts can automate anything. This is why DevOps and cloud automation love Linux.
  • 🌐
    Community: Millions of developers worldwide. Every problem has been solved and the solution is a Google search away.
Chapter 02 — Linux Commands

Speaking the Language of Linux

The command line is not a relic of the past. It is the most powerful interface ever built for controlling a computer. Every cloud operation, every automation script, every server configuration starts here.

📖 THE MENTAL MODEL

Think of the Linux shell like having a conversation. You type a command (a request), the kernel processes it, and the shell prints the result. Commands follow a pattern: command [options] [arguments]. The command is what to do, options are how to do it, and arguments are what to do it on.

For example: ls -la /home/user — ls is the command (list), -la are options (long format + show hidden files), and /home/user is the argument (list this directory). Once you see this pattern, every command makes sense.

Navigating the File System

Linux organizes everything in a single tree starting from / (root). There are no drive letters like C: or D: — everything hangs off root. Your files live in /home/yourname, system programs in /usr/bin, configuration in /etc, and temporary files in /tmp.

$ pwd # Print Working Directory — where am I right now? /home/student $ ls # List files in current directory $ ls -l # Long format: permissions, owner, size, date $ ls -la # Long format + hidden files (those starting with .) $ ls -lh # Human readable sizes (KB, MB instead of bytes) $ cd /home/user # Change Directory to absolute path $ cd Documents # Change to relative path $ cd .. # Go up one level $ cd ~ # Go to your home directory $ cd - # Go back to previous directory

Working with Files and Directories

# Create $ mkdir mydir # Make a directory $ mkdir -p a/b/c # Create nested dirs in one shot (-p = parents) $ touch file.txt # Create empty file (or update timestamp) # The classic lab task: create 100 files in one command $ mkdir -p dbda/test && touch dbda/test/file{1..100} # Copy, Move, Remove $ cp source.txt dest.txt # Copy file $ cp -r sourcedir/ destdir/ # Copy directory recursively $ mv old.txt new.txt # Move/rename file $ rm file.txt # Remove file $ rm -rf directory/ # Remove directory and all contents (be careful!) $ rmdir emptydir/ # Remove empty directory only # Links $ ln file.txt hardlink.txt # Hard link — same file, two names $ ln -s file.txt symlink.txt # Symbolic link — like a shortcut/alias

Reading and Searching Files

$ cat file.txt # Print entire file to screen $ head -20 file.txt # First 20 lines $ tail -20 file.txt # Last 20 lines $ tail -f /var/log/syslog # Follow a log file in real-time (Ctrl+C to stop) $ grep "error" file.txt # Find lines containing "error" $ grep -i "error" file.txt # Case-insensitive search $ grep -r "error" /var/log/ # Recursive search in directory $ grep -n "error" file.txt # Show line numbers $ find /home -name "*.txt" # Find all .txt files under /home $ find / -user alice # Find all files owned by alice $ locate filename # Faster search using pre-built index $ sort names.txt # Sort lines alphabetically $ sort -n numbers.txt # Sort numerically $ diff file1.txt file2.txt # Show differences between two files

Compression and Archiving

In Linux, archiving (bundling files together) and compression (making them smaller) are separate operations. tar bundles files into a single archive. gzip compresses them. Often both are done together.

# tar — Tape ARchive $ tar -cvf archive.tar files/ # Create archive (c=create, v=verbose, f=file) $ tar -xvf archive.tar # Extract archive (x=extract) $ tar -cvzf archive.tar.gz files/ # Create + compress with gzip (z flag) $ tar -xvzf archive.tar.gz # Extract + decompress $ tar -tvf archive.tar # List contents without extracting # gzip / gunzip $ gzip bigfile.txt # Compress → creates bigfile.txt.gz, original deleted $ gunzip bigfile.txt.gz # Decompress # zip / unzip (compatible with Windows) $ zip archive.zip file1 file2 $ unzip archive.zip

Process Management

$ ps # List processes running in your current session $ ps aux # List ALL processes on the system $ ps aux | grep python # Find python processes specifically $ kill 1234 # Send SIGTERM (graceful stop) to process with PID 1234 $ kill -9 1234 # Send SIGKILL (force stop) — cannot be ignored $ who # Who is logged into the system $ whoami # Which user you are $ echo "hello" # Print text to screen

Remote Access Commands

These commands are critical for server administration — and for cloud computing, where you almost never have physical access to a machine.

Chapter 03 — File Permissions & Security

Who Gets to Touch What

Linux was designed from day one as a multi-user system. The permission model is not a feature bolt-on — it is the core of Linux security, and understanding it deeply is non-negotiable for any server work.

The Three-Layer Permission System

Every file and directory in Linux has three permission sets: one for the owner (the user who created it), one for the group (a set of users), and one for others (everyone else). Each set has three permission bits: read (r), write (w), and execute (x).

# When you run: ls -l -rwxr-xr-- 1 alice developers 4096 Jan 10 script.sh # - rwx r-x r-- # | | | | # | | | └── Others: read only # | | └── Group (developers): read + execute # | └── Owner (alice): read + write + execute # └── File type (- = file, d = directory, l = symlink)

chmod — Changing Permissions

You can set permissions two ways: symbolic (using letters) or numeric (using octal numbers). Both are important to know.

# Symbolic mode — readable but verbose $ chmod u+x script.sh # Add execute for user (owner) $ chmod g-w file.txt # Remove write for group $ chmod o+r file.txt # Add read for others $ chmod a+x script.sh # Add execute for ALL (u+g+o) # Numeric (octal) mode — compact and common # r=4, w=2, x=1. Add them: rwx=7, rw-=6, r-x=5, r--=4 $ chmod 755 script.sh # owner=rwx(7), group=r-x(5), others=r-x(5) $ chmod 644 file.txt # owner=rw-(6), group=r--(4), others=r--(4) $ chmod 600 private.key # owner=rw-(6), group=---(0), others=---(0) — SSH keys! # Change owner and group $ chown alice file.txt # Change owner to alice $ chown alice:developers file.txt # Change owner and group $ chgrp developers file.txt # Change group only

Access Control Lists (ACLs) — Fine-Grained Permissions

The traditional owner/group/others system works well, but sometimes it's not flexible enough. What if you want Alice to have read access, Bob to have read+write, but Charlie to have no access at all — while the file is owned by root and in the 'admin' group?

This is where ACLs (Access Control Lists) come in. ACLs let you set individual permissions for any specific user or group, independent of the file's owner and group.

# First, create the file $ touch /tmp/acl_file # Grant Larry and Curly: read + write + execute $ setfacl -m u:larry:rwx /tmp/acl_file $ setfacl -m u:curly:rwx /tmp/acl_file # Deny Moe all access $ setfacl -m u:moe:--- /tmp/acl_file # Allow group 'stooges' read+write (Moe's individual denial still applies) $ setfacl -m g:stooges:rw /tmp/acl_file # View the ACL $ getfacl /tmp/acl_file # Remove an ACL entry $ setfacl -x u:moe /tmp/acl_file # Remove ALL ACL entries $ setfacl -b /tmp/acl_file

ACL Priority Rule: When both a user-specific ACL and a group ACL apply, the most specific rule wins. A user-specific ACL for Moe (---, no access) overrides the group ACL giving stooges read+write. This is what makes ACLs powerful — fine-grained exceptions without changing the overall structure.

The vi/vim Editor

On servers (including cloud VMs), graphical editors don't exist. vi is always available, even on minimal systems. You must know it. The key insight: vi is a modal editor — it has different modes, and knowing which mode you're in is everything.

Chapter 04 — Shell Scripting

Automating the World with BASH

A shell script is a program. You write instructions, and the shell executes them one by one. This is how servers are configured, data pipelines are run, backups are scheduled, and cloud infrastructure is managed at scale.

📖 WHY SCRIPTING MATTERS FOR BIG DATA

Imagine you work at a company processing 10TB of log data every night. Someone has to write the script that starts the Spark job at 2 AM, checks if it succeeded, moves the output files to S3, sends an alert if anything failed, and cleans up temporary files. That "someone" is a shell script. Shell scripting is the glue that holds the entire data engineering world together.

Your First Script

#!/bin/bash # The shebang line ↑ tells the OS to use bash to run this script # Everything after # is a comment # Variables — no spaces around = NAME="Alice" AGE=25 GREETING="Hello, $NAME! You are $AGE years old." echo $GREETING # Reading input from user read -p "Enter your name: " USERNAME echo "Welcome, $USERNAME!" # Command line arguments # $0 = script name, $1 = first argument, $2 = second, $# = count echo "Script name: $0" echo "First arg: $1" echo "Total args: $#" # Arithmetic A=10 B=3 echo $((A + B)) # 13 — double parentheses for arithmetic echo $((A * B)) # 30 echo $((A / B)) # 3 (integer division)

Making Decisions — if/else

#!/bin/bash NUM=42 # Basic if-then-fi if [ $NUM -gt 40 ]; then echo "Greater than 40" fi # if-then-else-fi if [ $NUM -eq 42 ]; then echo "The answer to everything!" else echo "Just a number" fi # Numeric tests: -eq -ne -gt -ge -lt -le # String tests: = != -z (empty) -n (not empty) # File tests: -f (is file) -d (is directory) -e (exists) -r (readable) # File test example if [ -f /etc/passwd ]; then echo "/etc/passwd exists" fi # Case statement — cleaner than many if-elif DAY="Monday" case $DAY in "Monday") echo "Start of work week" ;; "Friday") echo "Almost weekend!" ;; "Saturday" | "Sunday") echo "Weekend!" ;; *) echo "Midweek" ;; esac

Loops

#!/bin/bash # for loop — iterate over a list for fruit in apple banana mango; do echo "Fruit: $fruit" done # for loop over range for i in {1..10}; do echo "Number $i" done # C-style for loop for ((i=0; i<5; i++)); do echo "i = $i" done # while loop COUNT=0 while [ $COUNT -lt 5 ]; do echo "Count: $COUNT" COUNT=$((COUNT + 1)) done # until loop — opposite of while (runs until condition is TRUE) until [ $COUNT -ge 10 ]; do COUNT=$((COUNT + 1)) done # break and continue work just like in other languages for i in {1..10}; do if [ $i -eq 5 ]; then continue # Skip 5 fi if [ $i -eq 8 ]; then break # Stop at 8 fi echo $i done

Chapter 05 — Git & GitHub

Never Lose Your Work Again

Version control is not optional in modern software development. Git is the system that tracks every change, who made it, and why — making collaboration, rollback, and deployment possible at scale.

📖 WHAT HAPPENS WITHOUT VERSION CONTROL

Imagine five developers editing the same Python file. Developer A makes a change. Developer B also makes a change — to the same file. They share it over email. Now there are two versions: analytics_v2_final.py and analytics_v2_final_bob_edit.py. Next week: analytics_v2_final_FINAL_use_this_one.py. A month later: nobody knows which version is production. A critical bug fix exists in only one version. This is hell. Git solves all of it.

How Git Actually Works

Git doesn't just save file copies. It takes snapshots of your entire project at each commit. It stores these efficiently — if a file hasn't changed, it just stores a reference to the previous snapshot. Every snapshot has a unique ID (a hash like a3f2c1d), an author, a timestamp, and a message.

GIT WORKFLOW — THE THREE AREAS
📁
Working Directory Where you actually edit files. Your normal file system. Changes here are "unstaged" — Git sees them as modified but hasn't tracked them yet.
📋
Staging Area (Index) A preparation area. You git add files here to say "include these changes in my next commit." You can stage only some changes from a file, giving you precise control over what goes into each commit.
🗄️
Local Repository (.git folder) The full history, stored locally. When you git commit, a permanent snapshot goes here. This is entirely on your machine — no internet needed.
☁️
Remote Repository (GitHub) The shared copy on the internet. git push sends your commits here. git pull brings others' commits to you. GitHub is the most popular host — it adds a web UI, pull requests, and collaboration features on top of raw Git.
# One-time setup $ git config --global user.name "Your Name" $ git config --global user.email "you@example.com" # Start a new project $ git init # Initialize a new repo in current directory $ git clone https://github.com/user/repo.git # Clone an existing repo # Daily workflow $ git status # See what's changed $ git add filename.py # Stage a specific file $ git add . # Stage ALL changes $ git commit -m "Add data validation function" # Commit with message $ git push origin main # Push to GitHub $ git pull # Get latest from GitHub # Viewing history $ git log # Full history $ git log --oneline # Compact view $ git diff # See what changed since last commit # Branching — work on features without breaking main $ git branch feature-x # Create branch $ git checkout feature-x # Switch to it $ git merge feature-x # Merge back to main
☁️ AWS REAL-WORLD CONNECTION

In the lab, you'll deploy an application to AWS directly from GitHub. AWS CodePipeline watches your GitHub repository — the moment you git push to the main branch, it automatically pulls your code, runs tests, and deploys to your EC2 instance or Lambda function. This is CI/CD (Continuous Integration / Continuous Deployment) — the modern way software reaches production. No manual uploads. No FTP. Just: git push → your app is live.

Chapter 06 — Introduction to Cloud Computing

What the Cloud Actually Is

The cloud is not magic, and it's not just "someone else's computer" — though that's a good starting point. It's a radically different way of thinking about computing resources, and it has changed every industry on Earth.

📖 THE PROBLEM THAT CREATED CLOUD COMPUTING

Imagine you're starting an e-commerce company in 2005. Before you can sell a single product, you must: buy servers (₹50 lakhs minimum), find a data center to host them, hire people to manage them, buy networking equipment, set up cooling and power, and wait 3 months for everything to arrive. Then, on launch day, if 10x more people visit than expected — your servers crash. On quiet days — your expensive servers sit idle at 5% capacity.

You're paying for peak capacity even on average days. You're paying for hardware that takes months to provision. This is insane for a business. Cloud computing was built to solve exactly this.

The Five Essential Characteristics of Cloud

NIST (National Institute of Standards and Technology) defines cloud computing by five characteristics. If something doesn't have all five, it's not really "cloud" — it's just hosting.

THE FIVE PILLARS OF CLOUD COMPUTING
📱
1. On-Demand Self-Service You provision resources yourself, instantly, without talking to a human. Log into AWS, click "Launch Instance," and in 90 seconds you have a server. No procurement form, no 3-month wait, no IT ticket. The entire process takes less time than making a cup of chai.
🌐
2. Broad Network Access Resources are available over the network and accessible from any device — laptop, phone, tablet — from anywhere in the world. Your AWS EC2 instance in Mumbai is reachable from your home in Pune, your office in Bangalore, or a coffee shop in London.
🏊
3. Resource Pooling The provider's computing resources serve multiple customers (multi-tenancy) from a shared pool, dynamically assigned based on demand. You don't know or care which physical server your VM runs on. AWS has millions of servers — they're shared intelligently across millions of customers.
📈
4. Rapid Elasticity Resources can be scaled out (add more) or in (remove some) rapidly — automatically if needed. Your application can handle 100 users at 3 AM and 1 million users at 3 PM without any manual intervention. This is the most game-changing characteristic for business.
📊
5. Measured Service Resource usage is monitored, controlled, and reported. You pay only for what you use — like your electricity bill. 10 hours of a server running costs exactly 10× the hourly rate. You use 500 GB of storage, you pay for 500 GB. No flat fees for unused capacity.

The Big Three Cloud Vendors

MARKET LEADER
AWS

Amazon Web Services. Launched 2006 — the original cloud. Largest market share (~33%). Most mature services, largest community, most job postings. If you learn one cloud, learn this one.

ENTERPRISE FAVOURITE
Azure

Microsoft Azure. Dominant in enterprises already using Microsoft stack (Office 365, Active Directory, SQL Server). Deep hybrid cloud support for companies transitioning from on-premise.

DATA & AI POWERHOUSE
GCP

Google Cloud Platform. Strongest in data analytics (BigQuery), machine learning (Vertex AI), and Kubernetes (they invented it). Preferred by data engineering and ML teams.

KEY INSIGHT

All three clouds offer essentially the same core services (VMs, storage, databases, functions). The difference is in ecosystem depth, pricing, support, and specific strengths. Most large companies use 2–3 clouds (multi-cloud strategy) to avoid vendor lock-in. In this course, we'll use AWS as our primary example with Azure as a secondary platform.

Chapter 07 — Cloud Service Models

SaaS, PaaS, IaaS — What You Manage

These three letters are the most important framework for understanding cloud services. They define how much of the stack the cloud vendor manages — and how much is your responsibility.

📖 THE PIZZA ANALOGY

Think of computing like making pizza. On-premise: you buy a pizza oven, make dough from scratch, source ingredients, bake everything yourself — total control, total effort. IaaS: you rent a commercial kitchen with ovens already installed — you still make the pizza, but you didn't buy the oven. PaaS: you rent a kitchen where the dough is already made — you just add toppings and bake. SaaS: you order Domino's — you don't even see the kitchen. Each step: less control, less effort, more convenience.

FULLY MANAGED
SaaS

Software as a Service. A complete application delivered over the internet. The vendor manages everything — servers, OS, middleware, the application itself, updates, backups.

You manage: Only your data and user settings.

Examples: Gmail, Salesforce, Slack, Zoom, Microsoft 365, Google Workspace
PLATFORM MANAGED
PaaS

Platform as a Service. A development platform where you deploy your code, and the cloud manages the OS, runtime, scaling, and infrastructure automatically.

You manage: Your application code and data.

Examples: AWS Elastic Beanstalk, Heroku, Google App Engine, Azure App Service
INFRASTRUCTURE MANAGED
IaaS

Infrastructure as a Service. Raw computing infrastructure — virtual machines, storage, networking. You get a machine; you're responsible for everything above the hardware.

You manage: OS, runtime, applications, data, security patches.

Examples: AWS EC2, Azure VMs, Google Compute Engine, DigitalOcean Droplets

SaaS in Detail — Why It Changed Everything

Before SaaS, software was a product you bought on a CD, installed on your machine, and that was it. When version 2.0 came out, you bought another CD. No updates unless you paid again. No collaboration. Works on one machine only.

SaaS changed this model fundamentally. With Gmail, there's no "install Gmail." It's just there, in your browser, always on the latest version, accessible from any device, with 15 GB of storage that lives in Google's cloud. When Google adds a feature, you have it instantly. When Google upgrades their servers, your Gmail gets faster automatically.

For businesses, the SaaS model means: no software licenses to buy, no servers to maintain, monthly/annual subscription pricing (operational expense, not capital expense), automatic updates, and access from anywhere. This is why companies like Salesforce, valued at hundreds of billions of dollars, sell software that requires you to install absolutely nothing.

IaaS in Detail — The Foundation of Cloud

IaaS is the base layer — raw compute, storage, and networking. Think of it as the data center without the physical building. AWS EC2 is the defining example of IaaS. You launch a virtual machine, choose its size (2 CPUs + 8GB RAM? 96 CPUs + 768GB RAM?), pick an operating system, and 90 seconds later you have a fully functional server.

☁️ AWS EC2 EXAMPLE — IaaS IN ACTION

When you launch an EC2 instance on AWS: you choose an AMI (Amazon Machine Image) — a snapshot of an OS (Ubuntu 22.04, Amazon Linux, Windows Server). You choose an instance type — t3.micro (2 vCPU, 1GB RAM) for small tasks, or c7g.48xlarge (192 vCPU, 384 GB RAM) for serious compute. You set up Security Groups — firewall rules controlling which ports accept traffic. You attach EBS volumes for persistent disk storage. And you pay only while the instance is running — stop it, and billing stops.

This is IaaS: the hardware, network, and virtualization are AWS's problem. The OS configuration, software installation, application deployment, security patching — that's your job.

Chapter 08 — Virtualization

The Technology That Makes Cloud Possible

Cloud computing is built on virtualization. Without it, you couldn't have hundreds of customers sharing the same physical server, or launch a new VM in 90 seconds. Understanding virtualization is understanding the engine of the cloud.

What is Virtualization?

Virtualization is the creation of a virtual (software-defined) version of something — a server, storage device, or network — that behaves exactly like the real thing. A Virtual Machine (VM) is a software emulation of a complete computer, including CPU, RAM, disk, and network interfaces.

The magic ingredient is the Hypervisor — software that runs on the physical host machine and creates, manages, and isolates VMs. It intercepts all hardware access requests from VMs and manages them so multiple VMs can safely share one physical machine without interfering with each other.

Thinks it has its own server. Has no idea other VMs exist.
🖥️

Completely isolated from VM 1. Different OS, same machine.
🖥️

Types of Hypervisors

TYPE 1 — BARE METAL
Native Hypervisor

Runs directly on hardware. No host OS underneath. Most efficient and performant. Used in all enterprise cloud environments.

Examples: VMware ESXi, Microsoft Hyper-V, KVM (used by AWS), Xen (used by many cloud providers).

TYPE 2 — HOSTED
Hosted Hypervisor

Runs on top of a host OS. Easier to use but less efficient. Good for personal development and testing on your laptop.

Examples: VirtualBox (Oracle), VMware Workstation, Parallels (Mac). Good for DBDA students to experiment with Linux locally.

Virtual Machine Migration — Live Migration

One of the most powerful capabilities of virtualization is live migration — moving a running VM from one physical server to another with zero downtime. This allows cloud providers to do maintenance on physical hardware without disrupting customer workloads.

AWS uses live migration constantly. When a physical server needs maintenance, AWS migrates all running EC2 instances to other physical servers transparently. From your VM's perspective, nothing happened. This is why cloud providers can guarantee 99.99% uptime — hardware failures don't cause outages because VMs are automatically migrated.

Chapter 09 — Administering Cloud Services

Monitoring, Pricing, and the Shared Responsibility Model

Launching a cloud resource is the easy part. The real skill is keeping it running efficiently, securely, and cost-effectively. Cloud administration is a full-time discipline.

The Shared Responsibility Model — Who Secures What?

This is one of the most important concepts in cloud computing, and it's also the source of most cloud security breaches. The cloud provider is responsible for the security of the cloud. You are responsible for the security in the cloud. The line between these two is called the Shared Responsibility Model.

SHARED RESPONSIBILITY — AWS EXAMPLE
👤
YOUR RESPONSIBILITY (Security IN the Cloud) Your data · Your OS patches · Your app security · IAM user permissions · Security group rules · Encryption configuration
You own this.
☁️
AWS RESPONSIBILITY (Security OF the Cloud) Physical servers · Network hardware · Data center access control · Hypervisor security · Base storage/compute/networking infrastructure
AWS owns this.
WHY THIS CAUSES BREACHES

Most cloud security incidents are not AWS's fault — they're the customer's. Someone leaves an S3 bucket set to public. Someone gives an IAM user admin permissions they don't need. Someone doesn't patch their EC2 instance's OS for 6 months. The cloud is as secure as you configure it to be. AWS secures the hardware; your job is to secure everything above the hypervisor.

Cloud Pricing — You Only Pay for What You Use

Cloud pricing is fundamentally different from buying hardware. Instead of large upfront capital expenditure, you pay small ongoing operational costs. The main pricing dimensions in AWS:

  • ⚙️
    Compute (EC2): Charged per second (for Linux) or per hour (for Windows) while the instance is running. Stop the instance — billing stops. An m5.large costs about $0.096/hour. Run it 24/7 for a month = ~$70. Run it only during business hours (10 hrs/day) = ~$29.
  • 💾
    Storage (S3, EBS): Charged per GB per month. S3 standard storage = ~$0.023/GB/month. If you store 1 TB of data, you pay ~$23/month regardless of whether anyone downloads it.
  • 🌐
    Data Transfer: Incoming data (ingress) is free. Outgoing data (egress) — sending data from AWS to the internet — costs money. ~$0.09/GB. This catches many beginners off guard.
  • 📞
    API Calls: Some services charge per API call. Lambda charges per invocation + duration. DynamoDB charges per read/write unit. This is how you can have "serverless" services with no minimum cost.

Cloud Monitoring — Knowing What's Happening

In a data center, you can walk up to a server and see if it's on. In the cloud, you need monitoring tools. AWS provides CloudWatch — a monitoring service that collects metrics (CPU usage, memory, network traffic, disk I/O), logs from your applications and services, and can trigger alerts or even automatic actions when metrics cross thresholds.

☁️ AWS CLOUDWATCH EXAMPLE

You set up a CloudWatch alarm: "If the CPU utilization of my EC2 instance exceeds 80% for more than 5 minutes, send me an email AND automatically add two more EC2 instances to my Auto Scaling group." This is automated scaling — your application handles a traffic surge automatically, without you waking up at 3 AM to manually add servers. When traffic drops, CloudWatch triggers another rule to remove the extra instances to save cost. This is the cloud's elastic nature in action.

Chapter 10 — AWS Deep Dive

Amazon Web Services — The Full Picture

AWS is not one service. It's over 200 services that work together to cover every possible computing need. You don't need to know all of them — but you must deeply understand the core ones.

EC2 — Elastic Cloud Compute

EC2 is the heart of AWS. It provides virtual machines — called instances — on demand. "Elastic" means you can scale the count up or down based on demand. Understanding EC2 means understanding several key concepts:

LAUNCHING AN EC2 INSTANCE — WHAT HAPPENS
📸
1. Choose an AMI (Amazon Machine Image) A pre-built OS snapshot. Choose Ubuntu 22.04 LTS, Amazon Linux 2023, or Windows Server 2022. The AMI determines your base OS. You can also create custom AMIs from your configured instances — like saving a checkpoint.
📦
2. Choose an Instance Type Instance types define how much CPU, RAM, and network capacity the VM gets. Types: General Purpose (t3, m6i), Compute Optimized (c7g — for CPU-heavy workloads), Memory Optimized (r6i — for in-memory databases), Storage Optimized (i3 — for massive disk I/O like Hadoop), GPU Instances (p4d — for deep learning).
🔑
3. Configure Key Pair AWS generates an RSA key pair. The public key is stored on the EC2 instance; you download the private key (.pem file). To SSH into your instance: ssh -i mykey.pem ec2-user@<public-ip>. Never share your .pem file — it's your password.
🔥
4. Set Up Security Groups Security Groups are virtual firewalls for your instance. You define inbound rules (what traffic can reach the instance) and outbound rules (what traffic can leave). To allow SSH: allow port 22 from your IP. To serve a web app: allow port 80 (HTTP) and 443 (HTTPS) from anywhere. Everything else is blocked by default.
💿
5. Attach EBS Storage EBS (Elastic Block Store) volumes are virtual hard drives that attach to your instance. They persist even if the instance stops (unlike instance store, which is temporary). Choose size (8 GB to 64 TB) and type (gp3 for general purpose, io2 for high-performance databases).
🚀
6. Launch — Ready in ~90 Seconds AWS provisions a physical machine, sets up the hypervisor, starts your VM, boots the OS, and assigns a public IP. You can now SSH in and do anything: install software, run code, host a web server, run a Spark job, process data.

AWS Lambda — Serverless Computing

Lambda is the opposite of EC2. With EC2, you manage a server. With Lambda, there is no server to manage. You just upload code (a function), and AWS runs it whenever it's triggered — handling all the infrastructure, scaling, and availability automatically.

Lambda is "serverless" — not that there's no server, but that you don't see, manage, or pay for a server when it's not doing work. You pay only per invocation and per millisecond of execution time. A Lambda that runs 1 million times for 200ms each costs about $0.20 — effectively free for many use cases.

# A simple AWS Lambda function in Python # This function runs when triggered (e.g., new file uploaded to S3) import json def lambda_handler(event, context): # 'event' contains information about what triggered the function # Could be: S3 file upload, API Gateway request, DynamoDB change, schedule bucket = event['Records'][0]['s3']['bucket']['name'] key = event['Records'][0]['s3']['object']['key'] print(f"New file uploaded: s3://{bucket}/{key}") # Process the file, transform data, send notification, etc. return { 'statusCode': 200, 'body': json.dumps('File processed successfully') }

S3 — Simple Storage Service

S3 is object storage — not a file system, not a database, but a massive key-value store where the key is an object's path (like data/2024/january/sales.csv) and the value is any file up to 5 TB in size. S3 stores trillions of objects and provides 99.999999999% (eleven nines) of durability — meaning if you store 10 million files in S3, you'd statistically lose less than one every 10,000 years.

  • 📁
    Buckets: Top-level containers (like folders), globally unique names. You create a bucket named "my-company-data" — no other AWS customer can use that name.
  • 📄
    Objects: Files stored in buckets. Each object has a key (path), data (the file), and metadata (content type, size, custom tags).
  • 🔐
    Access Control: By default, everything is private. You control access via bucket policies and ACLs. Public buckets are explicitly opted-in. This is a common misconfiguration that leads to data leaks.
  • 💰
    Storage Classes: S3 Standard (frequently accessed), S3-IA (infrequent access, cheaper), S3 Glacier (archival, very cheap but slow retrieval). Choose based on how often data is needed.

VPC — Virtual Private Cloud

A VPC is your own isolated section of the AWS cloud — a virtual network that you define and control. When you create an AWS account, you get a default VPC in each region. Inside a VPC, you create subnets (sub-networks) — some public (internet-accessible) and some private (internal only).

TYPICAL VPC ARCHITECTURE — PRODUCTION SETUP
🌐
Internet Gateway The door between your VPC and the public internet
↕
🟢
Public Subnet (10.0.1.0/24) Load Balancer · NAT Gateway · Bastion Host
Internet can reach these.
↕
🔵
Private Subnet — App Tier (10.0.2.0/24) EC2 instances running your web application servers
Internet cannot reach directly. Only via load balancer.
↕
🟡
Private Subnet — Data Tier (10.0.3.0/24) RDS Database · ElastiCache · S3 VPC Endpoint
Most isolated. Only app tier can reach this.

VPCs provide network-level isolation. Your EC2 instances in a private subnet cannot be reached from the internet at all — even if someone knows their IP addresses. They can only be reached from other resources inside your VPC. This is a fundamental security architecture pattern in every production cloud deployment.

Azure Storage — Three Types

Azure offers three distinct storage services for different use cases, all accessible from the same storage account:

  • 📦
    Blob Storage (Binary Large Object): Azure's equivalent of S3. Stores unstructured data — documents, images, videos, backups, big data files. Ideal for data lake scenarios where you store raw data in any format for later processing.
  • 📂
    File Storage: A fully managed file share accessible via the industry-standard SMB protocol. Think of it as a network drive in the cloud. Multiple VMs can mount and access the same file share simultaneously — useful for shared application configurations and legacy applications that need file system access.
  • 📬
    Queue Storage: A message queue for asynchronous communication between application components. One part of your application writes messages to the queue; another part reads and processes them at its own pace. Enables decoupling of system components for reliability and scalability.

Azure Functions — Serverless in Azure

Azure Functions is Azure's serverless compute service — equivalent to AWS Lambda. You write a function, define what triggers it (HTTP request, timer schedule, new blob in storage, new queue message), and Azure handles everything else.

🔷 AZURE DEVOPS — CI/CD IN PRACTICE

Azure DevOps is a complete platform for the entire software development lifecycle. It includes Azure Repos (Git hosting, like GitHub), Azure Pipelines (automated build, test, and deployment), Azure Boards (work item tracking), and Azure Artifacts (package management). In the lab, you'll deploy an application to Azure using Azure DevOps Pipelines — committing code triggers an automatic pipeline that builds your app, runs tests, and deploys to your Azure VM or App Service, with zero manual steps.

Azure Data Services for Big Data

Since this is a PGCP-BDA (Big Data Analytics) program, it's worth understanding how Azure specifically supports data workloads:

  • 🗄️
    Azure SQL Database: Fully managed relational database in the cloud. No server to manage. Automatic backups, high availability, and intelligent performance optimization built in. Pay for compute + storage separately.
  • 🌌
    Cosmos DB: Microsoft's globally distributed NoSQL database. Supports multiple data models (document, key-value, graph, column-family) through different APIs. Designed to serve data with single-digit millisecond latency from anywhere in the world simultaneously — useful for global real-time analytics applications.
  • 📊
    Azure Synapse Analytics: A unified analytics service combining big data processing (Spark) and enterprise data warehousing (SQL pools) into one platform. Similar to combining AWS EMR and Redshift.

Deploying AI and Analytics Workloads on Cloud

The cloud has become the natural home for AI and big data workloads because of three reasons: elastic compute (training a deep learning model might require 100 GPUs for 6 hours — cloud lets you do this and pay only for those 600 GPU-hours), managed services (both AWS SageMaker and Azure Machine Learning provide managed environments for training and deploying ML models without managing infrastructure), and data proximity (your training data is already in S3 or Blob Storage — keeping compute in the same cloud avoids massive data transfer costs).
